# Memex v4 最终设计文档

> 日期：2026-05-31
> 状态：最终定稿
> 技术路线：本地优先 · CLI 稳定契约 · MCP 永远 spawn CLI · Rust + Tauri 最终形态

---

## 1. 项目定位

Memex 是一个本地优先的跨 LLM 会话记忆中枢。

它自动采集 Cursor、Claude Code、Codex、OpenCode 等 AI 工具产生的本地会话历史，把这些历史整理成可读、可检索、可被 AI 调用的记忆层。

Memex 的核心价值是让 AI 和用户都能快速找回之前讨论过的方案、决策、错误、上下文和项目进展。

### 1.1 一句话说明

> Memex 让不同 AI 编辑器共享同一份本地会话记忆。

### 1.2 核心用户

- 频繁在 Cursor、Claude Code、Codex 等工具之间切换的工程师。
- 长周期项目开发者，需要找回几周或几个月前的 AI 讨论。
- 需要复盘项目决策、技术方案和历史上下文的团队成员。

### 1.3 产品边界

Memex 做：

- 本地 AI 会话采集。
- 会话清洗、脱敏、摘要和结构化。
- 跨工具全文检索。
- 通过 MCP 暴露给 AI 编辑器。
- 通过 Skill 提供显式触发入口。
- 通过 macOS menubar 提供用户可见入口。

Memex 不做：

- 不做任务管理。
- 不替代 IDE 自带历史。
- 不做公司知识库。
- 默认不做云端同步。
- 默认不上传用户会话内容到云端 LLM。

---

## 2. 设计目标

### 2.1 用户目标

- 用户可以搜索之前和 AI 讨论过的内容。
- 用户可以从搜索结果跳回完整 session。
- 用户可以看到今天或本周 AI 工具使用情况。
- 用户可以用 menubar 在不中断工作流的情况下搜索历史。

### 2.2 AI 目标

- AI 编辑器可以通过 MCP 自动调用 Memex 搜索历史。
- AI 可以基于历史上下文回答“之前讨论过什么”。
- AI 可以复用历史决策，减少重复解释和重复踩坑。

### 2.3 工程目标

- 本地优先，数据默认留在用户机器。
- Markdown 是主存储，SQLite 是可重建索引。
- CLI 是稳定契约层。
- MCP 永远只 spawn CLI，不直接依赖 HTTP。
- daemon 和 HTTP 只是 CLI 的内部加速层。
- v0.3 最终形态使用 Rust 主体 + Tauri menubar。

## 3. 最终架构

### 3.1 总体架构

```text
AI 工具本地会话
Cursor / Claude Code / Codex / OpenCode
        |
        v
Collector
增量读取本地会话文件
        |
        v
Processor
chunk / redact / summarize / metadata
        |
        v
Storage
Markdown 主存储 + SQLite / FTS5 派生索引
        |
        v
Retriever
chunk 级全文检索 + 本地语义检索 + 可选云端兜底
        |
        +------------------+------------------+------------------+
        |                  |                  |
        v                  v                  v
      CLI              Tauri menubar          HTTP API
稳定契约层             用户入口               内部加速与未来扩展
        |
        v
Rust MCP Server（memex mcp 子命令）
stdio 协议，直接调 memex-core，无 spawn 开销
        |
        v
AI 编辑器
```

### 3.1.1 增量处理主流程

一条本地会话从 AI 工具写入到可检索数据，按下面的顺序处理：

```text
AI 工具写入本地会话文件
        |
        v
Collector 收到文件变更
mtime / fs event / 手动 ingest
        |
        v
匹配 adapter
Claude Code / Cursor / Codex / OpenCode
        |
        v
从 last_processed_offset 开始解析
只读取新增字节或新增 JSONL 记录
        |
        v
生成 new_messages
统一 role / timestamp / content / tool_call
        |
        v
进入 Processor queue
按 session 顺序串行处理，同 session 不并发写
        |
        v
chunk / redact / summarize / metadata
先确定性处理，再按 LLM 配置增强
        |
        v
写 Markdown
可读主存储，失败则不推进 offset
        |
        v
写 SQLite + FTS5
结构化索引，可从 Markdown 重建
        |
        v
更新 offset
记录 adapter_state.last_processed_offset
```

这个顺序的关键点是：`offset` 最后更新。只要 Markdown 或 SQLite 写入失败，下次采集仍会从旧 offset 重放，避免丢消息。重复写入通过 `source_message_id` 和内容 hash 做幂等保护。

### 3.2 核心架构原则

#### 原则 1：Markdown 是真源

每个 session 都保存为一个 Markdown 文件。用户即使卸载 Memex，也能直接打开和阅读历史记录。

#### 原则 2：SQLite 是索引

SQLite 保存结构化元数据、chunk、FTS5 索引、摘要、访问日志和统计数据。SQLite 损坏时可以从 Markdown 重建。

#### 原则 3：CLI 是稳定契约

所有外部自动化入口都优先依赖 CLI。CLI 对外稳定，内部可以自动选择 HTTP 或 SQLite。

#### 原则 4：MCP 与 CLI 共用同一份 core

MCP Server 与 CLI 都是 `memex-core` 的薄包装。MCP 直接调用 core 函数，不感知 daemon、不感知 HTTP 端口；这样既保留"MCP 接口稳定"的契约，又避免 spawn 子进程的冷启动开销。如果 core 接口变更，MCP 与 CLI 同步演进。

#### 原则 5：云端能力必须显式授权

默认只使用本地检索和本地 Ollama。Claude fallback 属于云端能力，必须由用户显式开启。

## 4. 模块设计

### 4.1 Collector：采集层

Collector 负责从不同 AI 工具的本地会话目录中增量读取数据。

支持的数据源：

- Claude Code：`~/.claude/projects`
- Cursor：`~/Library/Application Support/Cursor/User/globalStorage/state.vscdb`（macOS，存于 SQLite blob，参考 tars-ai-butler 的 adapter 实现）
- Codex：`~/.codex/sessions`
- OpenCode：`~/.opencode`

核心能力：

- 按文件 mtime 检测变化。
- 按 offset 增量读取。
- 识别 session、message、role、timestamp。
- 过滤 private session。
- 记录 adapter 扫描状态。

### 4.2 Processor：处理层

Processor 负责把原始会话转成可存储、可检索的结构化内容。

处理流程：

```text
new_messages[]
      |
      v
normalize
把不同 adapter 的字段统一成 message envelope
      |
      v
chunk
按 turn、Markdown 结构、代码块、tool call、长日志切分
      |
      v
redact
生成 redacted_content 和 redaction 记录
      |
      v
metadata
用规则抽取 topics / languages / tools_used / error_keywords
      |
      v
summarize
可选增强：Ollama 本地优先，Claude opt-in 兜底
```

Processor 分成两类步骤：

- 确定性步骤：`normalize`、`chunk`、`redact`、`metadata`、`FTS5`。这些不依赖任何 LLM，必须永远执行。
- 增强步骤：`summarize`、`embedding`、云端 fallback。这些依赖 Ollama 或用户显式开启的 Claude，失败时不能阻塞确定性数据入库。

#### 4.2.1 chunk 定义

chunk 是 Memex 的最小检索单位。

一个原始 message 可能很长，里面可能同时包含自然语言、代码、tool call、错误日志和结论。如果直接把整条 message 当作检索单位，搜索结果会太粗，snippet 不准确，ranking 也容易被长日志污染。因此 Memex 会把 message 切成多个 chunk。

chunk 的切分原则：

- 按 Markdown 标题、段落和列表切分。
- 代码块独立成 chunk。
- tool call 独立成 chunk。
- tool result 或长日志按结构切分。
- assistant 的长回答可以按“解释、方案、代码、结论”拆分。
- chunk 保留原 message 的 `message_id`，方便回到完整上下文。

chunk 示例：

```json
{
  "id": 101,
  "message_id": 27,
  "chunk_type": "code",
  "content": "redis.pipeline().set(...).execute()",
  "position": 3,
  "token_count": 42,
  "metadata": {
    "language": "python",
    "topics": ["redis", "pipeline", "batch-write"]
  }
}
```

#### 4.2.2 redact 定义

redact 是脱敏步骤，用来在进入检索、摘要或云端 fallback 前移除敏感信息。

redact 不改变用户原始意图，只替换敏感片段。替换后保留类型占位符，让检索仍然知道这里出现过敏感信息。

脱敏前：

```text
The API key is sk-ant-123456 and the database host is db.internal.zoom.local.
```

脱敏后：

```text
The API key is [REDACTED:api_key] and the database host is [REDACTED:internal_host].
```

redact 规则来源：

- 内置规则：email、phone、API key、IP、银行卡。
- 上下文规则：`password is xxx`、`token: xxx`。
- 用户自定义规则：`~/.memex/redactions.yaml`。
- private session 规则：private 内容默认不进入 MCP 搜索结果。

redactions 表只保存 hash 和位置，不保存可还原原文。

#### 4.2.3 summarize 定义

summarize 是摘要步骤，用来把长会话压缩成更容易检索和复用的记忆。

摘要分四层：

| 层级 | 范围 | 作用 |
|---|---|---|
| L1 | chunk | 给单个 chunk 生成短摘要 |
| L2 | session | 总结一次完整会话 |
| L3 | project | 总结某个项目一段时间内的主题 |
| L4 | week | 生成周报或工作回顾 |

默认摘要路径：

```text
chunk content
      |
      v
Ollama 本地模型
      |
      v
summary 写入 chunks.summary 或 summaries 表
```

LLM 能力按下面的顺序判定：

1. Ollama 可用：生成 L1 chunk 摘要、L2 session 摘要和可选本地 embedding。
2. Ollama 不可用、Claude 未开启：跳过新摘要，只写 `summary_status = "pending_local_llm"`。
3. Ollama 不可用、Claude 已 opt-in：先对最小上下文脱敏，再调用 Claude 生成摘要或回答。
4. Ollama 和 Claude 都不可用：不生成新摘要，不生成 embedding，但 Markdown、messages、chunks、metadata、FTS5 仍正常写入。

无 LLM 时的降级结果示例：

```json
{
  "chunk_id": 101,
  "summary": null,
  "summary_status": "pending_local_llm",
  "embedding_status": "disabled",
  "searchable_by": ["fts5", "metadata", "time", "adapter", "project"]
}
```

这意味着用户仍然可以搜索 `redis pipeline`、按当前项目过滤、打开完整 session，只是看不到新生成的自然语言摘要和语义相似结果。

#### 4.2.4 metadata 定义

metadata 是从 message 或 chunk 中抽取出的结构化信息。

metadata 不是给用户直接阅读的正文，而是用于过滤、排序、聚合、统计和报告。

典型 metadata：

```json
{
  "topics": ["redis", "pipeline", "error-recovery"],
  "languages": ["python"],
  "has_code": true,
  "tools_used": ["read_file", "shell"],
  "error_keywords": ["timeout", "retry"],
  "sentiment": "neutral",
  "project": "memex"
}
```

metadata 的用途：

- 搜索时提升当前项目相关结果。
- 报告里聚合本周主题。
- menubar 中按 adapter、project、时间过滤。
- 识别错误踩坑和关键决策。
- 在没有 LLM 时，作为纯本地检索的补充信号。

#### 4.2.6 新会话生成物矩阵

新会话进入 Memex 后，先生成不依赖 LLM 的确定性数据，再按用户配置补充摘要和语义能力。

| 能力配置 | 一定生成 | 不生成 | 说明 |
|---|---|---|---|
| 无 Ollama，无 Claude | Markdown session、sessions、messages、chunks、redactions、metadata、chunks_fts、metrics | 新的 L1/L2/L3/L4 摘要、本地 embedding、Claude fallback 结果 | 系统仍可全文检索、按项目/时间/adapter 过滤、展示原文和基础 metadata。 |
| 有 Ollama，无 Claude | 上述全部，加 L1 chunk 摘要、L2 session 摘要、可选本地 embedding | Claude fallback 结果 | 默认推荐模式。所有摘要和语义索引都在本机生成。 |
| 无 Ollama，有 Claude 且用户 opt-in | 上述确定性数据，加已脱敏最小上下文的 Claude fallback 摘要或回答 | 本地 embedding、未授权云端调用 | 只在用户明确开启后使用，private session 默认不发送。 |
| 有 Ollama，有 Claude 且用户 opt-in | 本地确定性数据、本地摘要、本地语义索引、必要时 Claude fallback | 未脱敏云端上下文 | 检索顺序仍是 FTS5 -> Ollama -> Claude fallback。 |

没有任何 LLM 配置时，Memex 不是“空数据”。它只是缺少新摘要和语义增强，核心资产仍然是可读 Markdown、结构化消息、chunk 级 FTS 和 metadata。

#### 4.2.7 解析粒度

解析不是只对“当前最后一问”做，也不是先把整段 session 压成一个摘要再处理。Memex 的粒度分三层：

| 粒度 | 输入范围 | 产物 | 用途 |
|---|---|---|---|
| message | session 中每一次 user / assistant / tool 事件 | role、timestamp、原文、message_id | 保留原始上下文和跳转锚点。 |
| chunk | 单条 message 内按结构切出的片段 | chunk_type、content、position、token_count、L1 summary、metadata | 最小检索、snippet、ranking 单位。 |
| session | 当前 session 的全部 messages / chunks | L2 summary、关键决策、主题、风险、待办 | 详情页、报告和跨会话回顾。 |

增量采集时，Collector 从 `last_processed_offset` 之后解析新增 message。Processor 会为新增 message 生成 chunks，并按需更新 session 级摘要。也就是说：检索命中以 chunk 为单位，详情回看以 message 和 session 为单位，报告聚合以 session 和 project 为单位。

### 4.3 Storage：存储层

Storage 采用双存储设计。

```text
~/.memex/
├── config.toml
├── redactions.yaml
├── credentials.toml
├── index.db
├── sessions/
│   └── YYYY-MM-DD/
│       └── <adapter>_<topic>.md
├── reports/
├── logs/
├── cache/
└── backup/
```

写入顺序：

```text
Processor output
      |
      v
写 Markdown
      |
      v
写 SQLite
      |
      v
写 FTS5 index
```

`adapter_state` 示例：

```json
{
  "adapter": "cursor",
  "source_path": "/Users/example/.cursor/projects/memex/session.jsonl",
  "last_processed_offset": 18792,
  "last_processed_message_id": "cursor-msg-8f31",
  "last_seen_mtime": "2026-05-31T10:12:22+08:00",
  "status": "healthy",
  "last_error": null
}
```

当 Collector 再次收到同一个文件变化时，它从 `last_processed_offset` 后面继续读取。若源格式不是可按字节 offset 稳定追加的文件，则 adapter 可以把 offset 映射成 `line_number`、`event_index` 或源系统游标，但对 Collector 暴露的字段仍统一叫 `last_processed_offset`。

失败恢复：

- Markdown 写失败：重试后进入 dead letter queue。
- SQLite 写失败：保留 Markdown，后续通过 `memex rebuild-index` 重建。
- FTS5 损坏：从 `chunks` 表重建。

### 4.4 Retriever：检索层

Retriever 提供跨 session、跨 adapter 的统一搜索。

检索策略：

```text
query
  |
  v
L1: FTS5 BM25 chunk 级检索
  |
  +-- 命中：返回 top N
  |
  v
L2: Ollama 本地语义检索
  |
  +-- 命中：返回 top N
  |
  v
L3: Claude 云端兜底
仅用户 opt-in 后启用
```

排序信号：

- BM25 关键词相关度。
- 时间衰减。
- 当前 working directory 匹配。
- 同 session 相关性。
- 用户点击或使用频率。

返回字段：

- `id`
- `session_id`
- `adapter`
- `timestamp`
- `snippet`
- `summary`
- `match_reason`
- `url`

### 4.5 Exposure：暴露层

Exposure 提供 CLI、MCP、Skill、menubar、HTTP API 五类入口。

#### CLI

CLI 是最稳定的入口。

```bash
memex ingest
memex process
memex search "redis pipeline"
memex search --json "redis pipeline"
memex sessions list --days 7
memex sessions show <id>
memex stats
memex config get <key>
memex config set <key> <value>
memex rebuild-index
memex backup ~/memex-backup.tar.gz
```

#### MCP

MCP 是 AI 编辑器自动调用的入口。

MCP 提供三个工具：

- `search_memory`
- `get_session`
- `list_recent`

MCP 调用链：

```text
AI 编辑器
    |
    v
MCP call_tool
    |
    v
memex mcp 子命令（Rust，stdio）
    |
    v
直接调用 memex-core::retriever::search 等接口
    |
    v
MCP TextContent
    |
    v
AI 编辑器生成回答
```

由于 MCP server 与 CLI 共享同一个 `memex-core`，没有 spawn 子进程的开销，单次 tool call 端到端 < 25ms（不含 LLM 思考时间）。

#### Skill

Skill 是用户显式触发的入口。

典型触发语：

- “之前讨论过这个吗？”
- “查一下历史里 redis pipeline 怎么设计的”
- “找一下上次 Rust 迁移的决策”

Skill 内部仍然通过 MCP 调用 Memex。

#### menubar

menubar 是用户可见入口。

主要功能：

- 状态栏显示今日 AI 调用次数。
- 展示最近 session。
- 搜索历史会话。
- 查看 session 详情。
- 打开设置。
- 生成日报或周报。
- 配置 adapter、隐私、LLM 和备份。

#### HTTP API

HTTP API 只监听 `127.0.0.1`。

用途：

- daemon 存在时给 CLI 内部加速。
- v0.4+ 提供 Web UI 和第三方集成入口。

MCP 不直接调用 HTTP API。

## 7. 数据模型

### 7.1 核心表

| 表 | 用途 |
|---|---|
| `sources` | 数据源注册和扫描状态 |
| `sessions` | 会话元信息 |
| `messages` | 原始消息 |
| `chunks` | 消息切片 |
| `chunks_fts` | chunk 级全文检索 |
| `summaries` | 多层摘要 |
| `tags` | 标签 |
| `session_tags` | session 与 tag 关系 |
| `redactions` | 脱敏记录 |
| `access_log` | 访问日志 |
| `metrics` | 聚合统计 |
| `kv` | 配置和系统元数据 |

### 7.2 Markdown session schema

```markdown
---
schema_version: 1
session_id: claude-code_2026-05-28_rust-migration-001
adapter: claude-code
project: memex
working_dir: /Users/example/projects/memex
started_at: 2026-05-28T14:00:00Z
ended_at: 2026-05-28T15:30:00Z
message_count: 47
topics: [rust, sqlite, migration]
summary: 讨论 Cursor adapter 解析方案与 vscdb 字段映射
privacy: public
---

## Message 1 (user, 2026-05-28T14:00:00Z)

...

## Message 2 (assistant, 2026-05-28T14:01:23Z)

...
```

### 7.3 FTS 设计

FTS 建在 `chunks` 上，而不是建在完整 message 上。

原因：

- chunk 是实际检索和定位的最小单位。
- snippet 更准确。
- 长消息不会污染 ranking。
- tool output 和代码块可以独立命中。

## 9. 隐私与安全

### 9.1 默认行为

- 默认不上传会话内容。
- 默认不开启 Claude fallback。
- 默认只使用 FTS5 和本地 Ollama。
- private session 默认不进入 MCP 搜索。

### 9.2 云端 LLM opt-in

用户开启 Claude fallback 前，需要明确知道：

- 哪些内容会被发送。
- 内容会先 redaction。
- 只发送最小必要上下文。
- 可以随时关闭。

Claude fallback 的执行条件：

```text
用户开启 cloud_fallback
        |
        v
检测 ANTHROPIC_API_KEY 或 credentials.toml
        |
        +-- 未配置：提示用户配置，不调用云端
        |
        +-- 已配置：继续
        |
        v
FTS5 / Ollama 无足够结果
        |
        v
抽取最小必要上下文
        |
        v
执行 redaction
        |
        v
调用 Anthropic Claude API
        |
        v
返回补充结果并标记来源为 cloud_fallback
```

因此，Claude fallback 需要 API key。推荐优先从环境变量 `ANTHROPIC_API_KEY` 读取；如果用户希望长期保存，则写入本地 `credentials.toml`，并强制权限为 600 或 700。

### 9.3 无 LLM 降级策略

如果本地 Ollama 不可用，且用户没有开启或没有配置 Claude fallback，Memex 仍然保持可用。

可用能力：

- FTS5 BM25 全文检索。
- chunk 级 snippet。
- `match_reason`。
- adapter、project、时间过滤。
- 已经生成过的历史摘要。
- access_log 和 metrics。
- Markdown session 查看。
- MCP 通过 CLI 搜索历史。

暂停或降级的能力：

- 新 session 的 L1/L2/L3/L4 摘要生成暂停。
- 语义检索退化为关键词检索。
- 周报生成只能基于已有摘要和 metadata。
- 无法用 Claude 对低命中结果做语义补充。

推荐补救方式：

```text
Ollama 不可用 + Claude fallback 未配置
        |
        v
继续使用 FTS5 / metadata / 已有摘要
        |
        +-- 用户愿意本地模型：引导安装 Ollama
        |
        +-- 用户愿意云端兜底：引导配置 ANTHROPIC_API_KEY
        |
        +-- 用户都不愿意：保持纯本地关键词检索模式
```

这个降级路径不能阻断核心搜索功能。Memex 的最低可用形态是 `Markdown + SQLite + FTS5 + CLI`。

### 9.4 本地 HTTP 限制

- HTTP server 只监听 `127.0.0.1`。
- 默认端口 `9999`。
- 不暴露到局域网。
- MCP 不直接调用 HTTP。

### 9.5 凭证管理

- API key 不写入普通配置。
- `credentials.toml` 使用 600 或 700 权限。
- 支持从环境变量读取 `ANTHROPIC_API_KEY`。

## 10. 阶段计划

技术路线已收敛为"全程 Rust，无 Python 中间态"。具体 Sprint 拆分、checklist、风险与代码复用来源参见 `20260531-13-Memex-执行计划-Rust-70%.md`（文件名后缀为整体完成率），本章只给出阶段总览。

### 10.1 Phase 1 — Core MVP（Sprint 1 ~ 4，约 8 周）

CLI + 采集 + 存储 + 检索全部跑通。退出条件：4 个 adapter 中至少 3 个能采集真实数据；FTS5 搜索 < 50ms；backup → delete → rebuild 后数据一致。

### 10.2 Phase 2 — AI 暴露（Sprint 5 ~ 6，约 4 周）

Rust MCP Server（`memex mcp` 子命令）+ Cursor / Claude Code Skill + Ollama / Anthropic provider。退出条件：在 Cursor 与 Claude Code 中都能通过 MCP 调用 search 成功；MCP 启动到首次响应 < 25ms。

### 10.3 Phase 3 — Menubar（Sprint 7 ~ 9，约 6 周）

Tauri 2 menubar + axum daemon + file watcher + access_log。退出条件：menubar 打开 < 100ms；daemon 长期运行不漏消息；⌘⇧M 全局唤起可用。

### 10.4 Phase 4 — 打磨发布（Sprint 10 ~ 11，约 4 周）

Homebrew / DMG / launchd / 自动更新 / `memex doctor`。退出条件：性能压测达标；4 个 adapter 全部稳定；MCP 接口与 v0.2 兼容不破坏。
