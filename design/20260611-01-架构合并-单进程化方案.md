# 架构合并：把 memex-daemon / memex-cli / memex-mcp 收进 src-tauri 单进程

> **目的**：减少运行时进程数（当前 3 → 目标 1-2），消除 memex-core 重复静态链接（节省 ~10-15 MB），降低部署 / 运维心智负担。
> **范围**：仅分析与方案选择，不动代码。决策后另开一份实施计划文档。
> **日期**：2026-06-11

---

## 1. 现状盘点

### 1.1 4 个 crate 的角色与体积

| crate | 形态 | LOC | 依赖 memex-core | bundle 内 binary | binary 大小 |
|---|---|---:|:-:|---|---:|
| `memex-core` | lib | 18,889 | — | （静态链接进 3 个 binary） | — |
| `memex-mcp` | lib | 842 | ✓ | （静态链接进 memex-cli） | — |
| `memex-cli` | binary `memex` | 4,360 | ✓ | `Contents/MacOS/memex-cli` | 10.1 MB |
| `memex-daemon` | binary `memex-daemon` | 988 | ✓ | `Contents/MacOS/memex-daemon` | 10.9 MB |
| `memex-menubar` | Tauri lib + bin `Memex` | — | ✓ | `Contents/MacOS/Memex` | 20.2 MB |

合计 **41.2 MB**，其中 memex-core 静态链接 3 次重复 ~10-15 MB。

### 1.2 当前运行时进程模型

```
┌─────────────────────────────┐
│ Memex.app (GUI)             │
│   tauri runtime + webview   │
│   tray icon / window        │
│   PID 17399                 │ ← 用户感知的"Memex 在跑"
└──────────┬──────────────────┘
           │ Command::spawn (fork)
           │ HTTP localhost:<port>
           ↓
┌─────────────────────────────┐
│ memex-daemon                │ ← 真正干活的后台进程
│   axum HTTP server          │
│   inotify/FSEvents watcher  │
│   index updater (tokio)     │
│   日程任务                  │
│   PID 71485 (常驻)          │
└─────────────────────────────┘

┌─────────────────────────────┐
│ memex-cli (按需 spawn)      │
│   IDE stdio MCP server      │ ← Cursor/Claude Code/Codex/opencode
│   或单次 CLI 命令           │
│   生命周期跟随 IDE          │
└─────────────────────────────┘
```

3 进程，3 binary，3 份静态链接的 memex-core。

### 1.3 进程间通信

- **Tauri ↔ daemon**：HTTP（localhost），端口写在 `~/.memex/daemon.lock`
- **CLI ↔ memex-core**：直接打开 SQLite（**当前 memex-cli mcp 是冷启动直连 db，不走 daemon**）
- **Tauri ↔ CLI**：无直接通信（CLI 完全独立运行）

⚠️ 关键发现：**memex-cli mcp 不走 daemon**，每次 IDE 调用都冷启动 SQLite + 加载索引，几百 ms 开销重复支付。

### 1.4 痛点

| 痛点 | 影响 |
|---|---|
| 3 个 binary 各自静态链接 memex-core | ~10-15 MB 浪费磁盘 + 升级时要替换 3 个文件 |
| daemon 是独立进程 | 有 daemon.lock 锁文件、SIGTERM/SIGKILL 协议、孤儿进程 GC、日志路径协商等额外复杂度 |
| memex-cli mcp 冷启动直连 db | IDE 每次新会话调用都重新加载索引 + 占用 100+ MB 内存 |
| 用户看到 Menu Bar 旁边好几个 Memex 相关项 | （已修）但每个 binary 仍需独立部署 |
| 多个 binary 部署时 codesign / quarantine / xattr 要重复处理 | upgrade-local.sh / install-macos.sh 都有冗余清理逻辑 |

## 2. 合并方案矩阵

### A. 不动（基线）

- binary 数 3，进程 3，bundle 41 MB
- 改动 0
- 痛点全部保留

### B. 把 daemon 合进 Tauri 主进程（**推荐**）

```
┌─────────────────────────────────┐
│ Memex.app (单一长生命周期进程)  │
│   tauri runtime + webview       │
│   tray icon / window            │
│   ► 内嵌 axum HTTP server       │ ← 接管原 daemon 角色
│   ► tokio task: watcher         │
│   ► tokio task: indexer         │
│   ► tokio task: scheduler       │
│   memex-core Arc 单实例共享给   │
│   GUI commands & HTTP handlers  │
└─────────────────────────────────┘

┌─────────────────────────────┐
│ memex-cli (薄壳客户端)      │
│   IDE stdio MCP entry       │
│   通过 HTTP 转发给主进程    │ ← daemon 已不存在，CLI 直接连主进程
│   主进程没跑时退化为直连 db │
└─────────────────────────────┘
```

**收益**：
- 进程数 3 → 2（GUI+daemon 合并为 1，CLI 仍按需 spawn）
- binary 数 3 → 2（删除 memex-daemon binary）
- bundle 体积 41 → ~30 MB（少一份 memex-core 静态链接）
- 删除 daemon.lock / SIGTERM 协议 / 孤儿进程 GC / start_daemon_blocking 等约 200 行胶水
- IDE 调用 memex-cli mcp 可走 HTTP 复用主进程已加载的索引 → 冷启动 100ms → 10ms

**代价**：
- Tauri 主进程要承担 daemon 工作 → 内存 +30-60 MB（索引常驻）
- daemon 任务 panic 影响 GUI（要包好 catch_unwind / spawn 隔离）
- Tauri GUI 关闭主窗口时 daemon 任务必须**继续跑**（已是当前 menubar app 的预期行为，无新约束）
- CLI 还得保留独立直连 db 的 fallback 路径（主进程没启动时仍能用 `memex search` 这种独立命令）

**实施工作量**：6-10 小时
- 把 `memex-daemon/src/{routes,watcher,web,launchd,lockfile,logging}.rs` 挪进 `tauri-app/src-tauri/src/services/daemon/`
- Tauri builder.setup() 里启动 axum server + watcher tokio task
- 改写 `commands/daemon.rs`：从 spawn 子进程 + lock 文件协议 改成查 in-process state（`Arc<DaemonHandle>`）
- 删除 `memex-daemon` workspace member + Cargo `[[bin]]` + tauri.conf.json externalBin
- 改 `commands/cli_path.rs` 把 `TARGET_NAMES` 缩到只剩 `memex-cli`
- 改 `memex-cli/src/commands/mcp.rs`：增加 "尝试连主进程 HTTP / 失败 fallback 直连 db" 逻辑
- 更新 scripts/install-macos.sh / upgrade-local.sh / Casks/memex.rb
- 跑全 workspace `cargo test`

**风险**：
- ⚠️ axum tokio runtime 必须跟 tauri 内置的同一个（tauri 2 用 tokio，可以直接 `tauri::async_runtime::spawn`）
- ⚠️ daemon 退出方式：当前 Tauri 退出已走 `RunEvent::ExitRequested → stop_daemon_blocking`，合并后变成 drop in-process 任务，逻辑更简单
- ⚠️ daemon HTTP 端口管理：当前从 daemon.lock 读，合并后改成 Tauri builder 时申请、写入 `tauri::State`、暴露给 CLI 通过 fixed Unix socket 或文件中转

### C. 完全单 binary（CLI + daemon + menubar 全收进 Memex.app）

```
┌─────────────────────────────────┐
│ Memex (唯一 binary)             │
│   argv[1] == nil → GUI mode     │
│   argv[1] == "mcp" → CLI MCP    │
│   argv[1] == "context" → CLI    │
│   argv[1] == "daemon" → 子模式  │
│   …                             │
└─────────────────────────────────┘
```

**收益**：
- binary 数 3 → 1
- bundle 体积 41 → ~22 MB（只剩一份 memex-core）

**代价**：
- IDE 调用 `Contents/MacOS/Memex mcp` 时，Tauri 主入口 main() 会先初始化 webview / NSStatusItem / Info.plist binding 等 GUI runtime，再判断 argv 走 CLI 分支 → **每次 IDE 启动都额外承担 GUI runtime 初始化代价**（~500ms 冷启动）
- ⚠️ macOS bundle 规约：`Contents/MacOS/<CFBundleExecutable>` 启动时 OS 会做 LaunchServices 注册、Dock 反弹、quarantine 检查；同一个 binary 当 CLI stdio server 用会触发上述 OS 行为
- ⚠️ Tauri builder 内 NSStatusItem 创建必须早于事件循环，但 CLI 模式下用户没有"创建 status item"的预期 → 会产生 ghost menu bar entry
- bundle 主 binary 同时是 IDE 调用的 stdio server → IDE 写 mcp.json 时 command 字段配 `Memex.app/Contents/MacOS/Memex` → APFS case-insensitive 已不会撞名（CLI sidecar 不存在了），但是**进程名同样是 `Memex`** → ps 里"几个 Memex 进程"指代不清

**实施工作量**：12-20 小时（含 Tauri main 启动路径重构）

**风险**：
- 上面的 4 个 ⚠️ 都是真问题，非纸面分析
- 单 binary 当 GUI 和 CLI 双角色用 在 macOS 上属于 anti-pattern
- 收益（节省 1 个 binary）远不足以平衡这些复杂度

---

## 3. 推荐：方案 B

理由：
1. **可量化收益**：bundle 减 11 MB（27%），进程减 1（33%），cold-start MCP 调用快 10 倍
2. **复杂度可控**：axum HTTP server 已有完整实现，只是从独立 binary 搬进 Tauri 进程；CLI 改造为薄客户端是增量改动
3. **避开 macOS bundle 陷阱**：menubar GUI 仍独立 binary，CLI 仍独立 binary，APFS 大小写撞名、CFBundleExecutable / NSStatusItem ghost 这些都不会复发
4. **保留逃生通道**：CLI 仍能在主进程没跑时直连 db，单元测试、CI、命令行用户的体验不变

不推荐方案 C：单 binary 同时担 GUI + CLI 是 macOS bundle 反模式，今天已经踩过 APFS 大小写撞名的坑，再叠加 GUI runtime 跟 stdio CLI 共存只会让问题更糟。

## 4. 如果走方案 B，分阶段实施提纲

> 不在本文档落实施细节，仅给阶段切片让用户评估接受度。

| 阶段 | 内容 | 可独立合入 |
|---|---|:-:|
| Phase 1 | `memex-daemon` 改 lib（保留 `[[bin]]` 但 main.rs 只是 `lib::run()` 的薄包装） | ✓ |
| Phase 2 | Tauri 启动时 `tauri::async_runtime::spawn` 跑 daemon `lib::run()`，daemon 仍可独立部署也可内嵌 | ✓ |
| Phase 3 | 切换默认 daemon 启动方式从 fork-subprocess 到 in-process；保留环境变量 `MEMEX_DAEMON_EXTERNAL=1` 走旧路径 | ✓ |
| Phase 4 | 默认 in-process 稳定后删除 fork-subprocess 代码 + memex-daemon `[[bin]]` + sidecar | ✓ |
| Phase 5 | `memex-cli mcp` 增加 HTTP-first / db-fallback 双链路 | ✓ |
| Phase 6 | bundle 后处理：Cask / install scripts / upgrade-local 同步更新 | ✓ |

每个 phase 都是可独立 ship 的最小单元。

## 5. 决策点

需要用户回答的问题：

1. **接受方案 B**（GUI + daemon 合并，CLI 仍独立薄壳）吗？
2. 是否同意 Phase 1-4 分多个 commit / PR 推进？还是一次合 ?
3. 是否要保留 `MEMEX_DAEMON_EXTERNAL=1` 这种逃生开关 ?（推荐保留至少一个发布周期，方便回滚）
4. memex-cli 改薄壳后，是否还需要保留 `memex search` / `memex context` 等独立命令的离线能力（不依赖主进程 HTTP）？
5. 是否同步把 `memex-daemon` 作为独立 binary 的发布通路（如 brew formula、独立 daemon-only 镜像）一并废止？
