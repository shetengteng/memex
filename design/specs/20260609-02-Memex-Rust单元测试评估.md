# Memex Rust 单元测试评估

> 创建时间：2026-06-09
> 范围：`crates/memex-core`、`crates/memex-cli`、`crates/memex-daemon`、`crates/memex-mcp`、`tauri-app/src-tauri`
> 目标：评估当前 Rust 代码的测试健康度，明确下一步最值得补的单元测试与轻量集成测试。

---

## 结论

当前 Rust 测试健康度：**B+，核心库覆盖较扎实，边界层覆盖偏薄**。

本次实测结果：

- `cargo test --workspace --all-targets`：通过，约 **283** 个测试通过。
- `cargo test --workspace --doc`：通过，但实际可执行 doctest 为 **0**；`memex-mcp` 的 crate 示例为 `ignore`。

已有测试对 `memex-core` 的 collector、SQLite 存储、查询、FTS 检索、隐私脱敏、LLM 解析与 fallback、MCP 工具协议覆盖较好。主要风险集中在三类代码：

1. CLI 命令层大多只通过真实副作用间接验证，缺少可注入上下文后的单元测试。
2. Tauri command 多数是 thin wrapper，但涉及 `memex_dir()`、真实 DB、provider 选择、线程和事件 emit，直接单测不足。
3. 自动 ingest 编排、summary pipeline、daemon watcher/lifecycle 这类跨模块流程，现有测试更多覆盖底层函数，缺少稳定的 fake adapter / fake provider 流程测试。

## 覆盖较好的模块

### 1. 存储与查询

`memex-core::storage` 覆盖质量较高：

- schema/migration/index 有结构测试。
- session/message/chunk/summary/thread CRUD 有 in-memory SQLite 单测。
- FTS search、JSON contract、workload daily/heatmap/report 有明确回归测试。
- rebuild 有 parser 级测试，也有一个 e2e roundtrip。

当前存储层适合继续作为其他模块测试的稳定 fake backend。新增业务逻辑优先用 `Db::open_in_memory()` 搭测试，不需要真实 `~/.memex`。

### 2. Collector 解析器

Claude Code、Cursor、Codex、OpenCode、Aider、Cline、Continue Dev 都有不同程度的 parser/scan/collect 测试。Cursor SQLite backend 的测试尤其有价值，覆盖了 composer headers、offset、multi-folder workspace、permission denied 等边界。

后续新增 adapter 时，应按现有模式至少覆盖：

- 空目录 / 缺失文件返回空结果。
- 正常 transcript 解析。
- offset 增量采集。
- project path / title 推断。
- malformed input 不 panic。

### 3. LLM 输出解析与 fallback

`llm::summarize`、`llm::threads`、`llm::reflect` 对 JSON fence、缺字段、重复 index、fallback cluster、空名称过滤等场景覆盖较好。这里已经具备“LLM 输出不稳定也不拖垮本地数据”的基本回归网。

## 主要缺口

### P1：补自动 ingest 编排测试

高价值原因：`run_ingest` / `ingest_adapter` 是 collector、privacy、processor、storage、markdown、summary pipeline 的交汇点。底层函数虽有测试，但流程组合缺少专门单测。

建议新增测试：

- fake adapter 返回 1 个 session + 多条 message，验证 session、message、chunk、source offset、markdown 全部落库。
- fake adapter 返回重复 message，验证 dedup 后 message/chunk 计数正确。
- fake adapter 的 `collect` 报错，验证 `run_ingest` 不失败、adapter error metric 增加。
- private session 被跳过，验证不会写 session/message/source state。
- 超过 `MESSAGE_BATCH_SIZE` 时，验证 batch 后 max offset 正确。

建议插入点：

- `crates/memex-core/src/ingest/adapter.rs` 内部 tests，覆盖 `ingest_adapter`。
- `crates/memex-core/src/ingest/mod.rs` 或 sibling `tests.rs`，覆盖 `run_ingest` 的错误隔离和 metrics。

### P1：补 CLI 命令层测试

高价值原因：CLI 是用户主要入口，当前测试只有 `io` 和 hooks。`search.rs`、`sessions.rs`、`stats.rs`、`config.rs`、`backup.rs` 等命令有输出格式、daemon fallback、JSON 模式差异，容易被重构破坏。

建议先做最小改造，不要重构大面：

- 给 URL query 构造提取纯函数，测试空格、`&`、`=`、中文、adapter/project/chunk_type/after/before。
- 给 direct search 输出模型提取 formatter，测试 JSON 与 human 两种格式。
- 给 `sessions` / `stats` 命令加 `MEMEX_HOME=tempdir` 场景测试：DB 不存在、空 DB、有数据。
- 把 `run` 内部依赖的 `memex_dir()`、daemon port 探测、HTTP get 做轻量注入，避免单测必须启动 daemon。

建议插入点：

- `crates/memex-cli/src/commands/search.rs`
- `crates/memex-cli/src/commands/sessions.rs`
- `crates/memex-cli/src/commands/stats.rs`
- `crates/memex-cli/src/commands/config.rs`

### P1：补 Tauri command 行为测试

高价值原因：Tauri backend 是前端 IPC 边界，现有 contract 测试能防 DTO 形状变化，但很多 command 的行为没有直接覆盖。

建议新增测试：

- `sessions::list_recent`：DB 不存在返回空数组；有 DB 时分页参数生效。
- `sessions::get_session`：DB 不存在返回 `None`；存在时返回 message/detail。
- `sessions::retry_summary`：DB 不存在返回 `not_found`；无 provider 返回 backend error。
- `search::search_memex`：DB 不存在返回空结果或结构化错误；filter 字段传递正确。
- `config` commands：默认配置、set/toggle 后 roundtrip。
- `maintenance` commands：高风险 reset 操作至少覆盖“只允许 memex dir 内路径”的防护。

建议先把纯业务逻辑下沉为普通函数，`#[tauri::command]` 只保留参数接收和错误转换。这样测试无需构造 Tauri runtime。

### P2：补 daemon lifecycle / watcher 测试

当前 daemon HTTP handler 基本可测，但 watcher、真实进程生命周期、lockfile 竞争和端口占用场景仍偏薄。

建议新增：

- stale lock 清理后可启动。
- port already in use 返回明确错误。
- watcher debounce 只触发一次 ingest。
- shutdown 时 lockfile 清理。

这些不必都做单元测试；涉及真实 socket / process 的场景可放 `tests/*.rs` 并用 `tempfile` 隔离。

### P2：补 provider 网络错误边界

当前 provider 相关测试主要是 model/name/config/API root 这类纯函数。Ollama、OpenAI-compatible、Anthropic 的 HTTP 错误、空响应、非 JSON 响应、timeout 行为没有明显单测。

建议不要默认打真实网络。优先把 request builder / response parser 拆为纯函数并测试：

- 2xx 正常响应解析。
- 401/403/500 错误消息保留上下文。
- 空 body / malformed JSON 返回结构化错误。
- provider unavailable 时 selector fallback 正确。

## 建议补测清单

| 优先级 | 模块 | 测试目标 | 推荐类型 |
|---|---|---|---|
| P1 | `memex-core::ingest::adapter` | fake adapter happy path、dedup、batch offset、private skip、adapter error metric | 单元测试 |
| P1 | `memex-cli::commands::search` | daemon URL 构造、daemon 失败 fallback、JSON/human 输出 | 单元测试 |
| P1 | `tauri commands::sessions/search/config` | DB 缺失、有数据、provider 缺失、filter 传递 | 单元测试 |
| P1 | `memex-core::llm::* provider` | response parser / error parser | 单元测试 |
| P2 | `memex-daemon` | watcher debounce、lock 竞争、port 占用、shutdown cleanup | 集成测试 |
| P2 | `retriever` | CJK、特殊符号、日期 filter、rank boost 的组合矩阵 | 参数化单元测试 |
| P2 | `collector::*` | malformed / partial transcript 的 property-like 样本 | 单元测试 |
| P3 | public API rustdoc | 给核心 public API 增加可运行 doctest 示例 | doctest |

## 不建议优先做的测试

- 不建议把 Ollama / OpenAI / Anthropic 真实请求纳入默认 `cargo test`，会引入网络、token、模型可用性不确定性。
- 不建议为了测试 Tauri UI 行为在 Rust 单元测试里强构完整 webview runtime；IPC DTO contract 和 command 纯函数测试更稳定。
- 不建议立刻追求行覆盖率百分比。当前项目风险更在跨模块流程和边界错误路径，先补高价值回归用例。

## 质量门禁建议

每次 Rust 代码变更至少运行：

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace --all-targets
```

涉及 public API 或 rustdoc 示例时额外运行：

```bash
cargo test --workspace --doc
```

涉及 Tauri IPC DTO 或前端调用契约时额外运行：

```bash
cd tauri-app && npx vitest run
```

## 下一步推荐

建议先做一轮 **P1 ingest + CLI search + Tauri sessions/search/config** 的小批量补测，目标不是增加很多测试数量，而是把“用户入口”和“自动采集入口”两条最容易回归的路径先锁住。

验收标准：

- 新增测试不依赖真实 `~/.memex`、真实网络或真实 daemon 进程。
- 每个新增测试使用 `tempfile` / `Db::open_in_memory()` 隔离状态。
- 默认 `cargo test --workspace --all-targets` 仍在本地快速通过。
- 失败断言验证具体错误或结果字段，不只判断 `is_err()` / `is_ok()`。
